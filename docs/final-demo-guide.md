# 最终录屏指南

这份文档用于最终提交和现场演示。目标不是把所有实现细节讲完，而是在 5 到 8 分钟内证明：项目完整覆盖题目三要求，并且不是一次性 Demo，而是一个可运行、可校验、可继续打磨的小说转剧本工作台。

## 录屏前检查

在录屏前执行：

```bash
npm install
npm test
npm run lint
npm run build
npm run dev
```

浏览器打开：

```text
http://localhost:3001
```

如果 3001 端口被占用，先关闭其他 Next.js 项目，或确认当前打开的是本项目。

## 演示主线

一句话主线：

```text
导入 3 章以上小说
↓
章节解析与大纲预览
↓
配置 OpenAI-compatible 模型
↓
生成结构化 YAML 剧本
↓
Schema 校验 + 质量清单
↓
保存到项目库 / 本地草稿
↓
导出 YAML
↓
展示 Schema 文档
```

## 推荐演示步骤

### 1. 开场，点题

打开首页或工作台，说明选择的是“题目三：AI 小说转剧本工具”。

要讲清楚三点：

- 输入是 3 个章节以上的小说文本。
- 输出不是普通文本，而是结构化 YAML 剧本。
- 作者可以继续编辑、校验、保存和导出。

### 2. 导入小说

点击“导入文本”，选择：

```text
samples/novel-3chapters.txt
```

如果现场文件选择不方便，点击“加载样例”。样例内容与 `samples/novel-3chapters.txt` / `samples/novel-3chapters.md` 是同一类长文本，足够覆盖 3 章以上要求。

讲解点：

- 支持 `.txt` / `.md`。
- 文件在浏览器本地读取。
- 导入后标题和正文进入工作台。

### 3. 展示章节大纲

指出页面已经识别出：

- 章节数量。
- 每章标题。
- 每章字数。
- 正文预览。

讲解点：转换前先暴露解析结果，用户知道系统到底读到了哪些章节，不是黑盒生成。

### 4. 展示模型配置

展示“模型配置”区域：

- 项目只维护一套 OpenAI-compatible AI 配置。
- 可以填写 base URL 和 API Key。
- 可以点击“获取模型”临时拉取模型列表，选择更便宜的 model。
- 模型列表只用于本次选择，不做多供应商、多模型管理后台。

录屏时不要展示完整 API Key。生产环境使用服务端配置，本地开发才需要在页面填调试配置。

### 5. 转换为 YAML 剧本

点击“转换为 YAML 剧本”。

展示右侧或“剧本审查”区域中 YAML 的四个顶层字段：

- `metadata`
- `characters`
- `scenes`
- `summary`

讲解点：这证明输出是结构化剧本，不是普通故事摘要。

### 6. 展示校验与质量清单

展示：

- Schema 校验通过。
- 剧本质量清单通过。
- 转换报告里的章节数、角色数、场景数、台词数。

讲解点：

- 缺字段不会用空字符串兜底。
- 坏 YAML 不能导出。
- 质量清单是结构交付检查，不是主观文学评分。

### 7. 展示入库能力

点击“保存到项目库”，然后进入“项目库”。

展示项目卡片上的：

- 项目标题。
- 最近生成状态。
- 最近使用的模型。
- 最近生成记录。

讲解点：这不是为了入库而入库。入库能力在界面上可见，作者能从项目库继续找回项目、查看最近生成记录、继续打磨剧本。

### 8. 展示本地草稿

回到工作台，点击“保存为新草稿”。

刷新页面后点击“加载”，展示：

- 标题恢复。
- 小说正文恢复。
- YAML 恢复。
- 转换报告恢复。

再点击“删除”，说明删除草稿不会清空当前编辑区。

### 9. 展示失败路径

手动删除 YAML 中的：

```yaml
metadata:
  title: ...
```

或者只删除 `title` 这一行。

展示：

- Schema 校验失败。
- 错误路径定位到 `metadata.title`。
- 质量清单出现失败项。
- “导出 YAML”不可用。

恢复字段后展示校验通过。

### 10. 导出与 Schema 文档

点击“导出 YAML”。

最后打开：

```text
docs/script-yaml-schema.md
```

说明文档中已经定义 YAML Schema，并解释了为什么使用 `metadata`、`characters`、`scenes`、`dialogue`、`summary` 这些结构。

## 5 分钟口播节奏

| 时间 | 内容 | 目标 |
| --- | --- | --- |
| 0:00 - 0:30 | 点题和项目目标 | 让评审知道你做的是题目三 |
| 0:30 - 1:20 | 导入小说和章节大纲 | 证明支持 3 章以上输入 |
| 1:20 - 2:00 | 模型配置 | 证明是真 AI 能力入口，不是死样例 |
| 2:00 - 3:00 | 生成 YAML | 证明输出结构化剧本 |
| 3:00 - 3:50 | Schema 校验和质量清单 | 证明可校验、可编辑 |
| 3:50 - 4:30 | 项目库 / 草稿 | 证明能继续打磨，不是一次性页面 |
| 4:30 - 5:00 | 导出和 Schema 文档 | 对齐题目文档要求 |

## 如果真实 AI 现场失败

不要硬装成功。直接切换到稳定路径：

1. 展示错误信息。
2. 说明系统会拒绝坏输出，不会兜底生成假 YAML。
3. 打开 `samples/output.yaml` 或点击“加载样例”展示已通过 Schema 的样例输出。
4. 继续展示编辑、校验、项目库和导出流程。

这样反而能证明系统有边界：AI 失败时不会把坏结果交给作者。

## 不要演示的东西

- 不要展示完整 API Key。
- 不要打开无关目录、账号或配置文件。
- 不要临时修改代码。
- 不要临时新建数据库表。
- 不要把测试用确定性转换器说成产品 AI 能力。
- 不要讲多供应商管理后台；当前设计就是唯一 AI 配置。

## 最终收束句

本项目完成了题目三要求：把 3 个章节以上小说文本转换成结构化 YAML 剧本，并提供 Schema 文档、实时校验、质量清单、项目保存和导出能力。作者拿到的不是一段不可控的 AI 文本，而是一份能继续编辑和打磨的剧本初稿。
