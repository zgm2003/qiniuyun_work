# Future TODO / Roadmap

当前版本已经完成比赛演示闭环：能导入小说、识别章节、生成 YAML 剧本、校验结构、保存草稿并导出。后续不应该继续随便堆功能，而是围绕三个真实问题推进：

1. 作者能不能长期管理自己的改编项目？
2. 生成结果能不能持续变好，而不是每次碰运气？
3. 系统能不能从 Demo 过渡到可运营产品？

## P1：下一阶段最值得做的产品增强

### 1. 场景级局部重写

**目标**：用户可以选择某个 scene，让 AI 只重写这一场，而不是整篇重来。

**价值**：作者不会一次满意，他需要局部打磨。这比继续做大而泛的功能更接近真实创作流程。

**建议 PR 拆分**：

- `feat: select scene for revision`
- `feat: add scene rewrite instruction`
- `feat: validate rewritten scene before merge`

**注意**：重写后的 scene 必须重新通过 Schema 校验，不能直接拼回 YAML。

### 2. 可解释转换质量评分

**目标**：给每次生成结果一个可解释评分，不是主观“好不好看”，而是结构质量。

**评分维度**：

- 章节覆盖率。
- 角色一致性。
- 场景完整度。
- 台词占比。
- Schema 通过率。
- 改编建议数量。

**建议 PR 拆分**：

- `feat: add conversion quality metrics`
- `feat: display coverage and dialogue ratio`
- `feat: add actionable improvement suggestions`

### 3. 多格式导出

**目标**：不只导出 YAML，还能导出作者更熟悉的格式。

**优先级**：

1. Markdown 剧本。
2. Fountain。
3. 分镜表格 CSV。
4. PDF。

**原则**：先从 YAML 生成其他格式，不要反过来维护多套数据结构。

### 4. 导入格式扩展

**目标**：在 `.txt` / `.md` 之外支持更多作者常用文件。

**建议顺序**：

1. `.docx`
2. `.pdf`
3. 多文件章节批量导入

**注意**：这会引入解析库和格式异常，比赛阶段不建议继续扩。

## P2：产品化能力

### 1. 登录系统

**目标**：用户可以拥有自己的项目、模型配置和历史记录。

**建议能力**：

- 邮箱 / GitHub 登录。
- Session 管理。
- 用户级项目隔离。
- 用户级 API Key 或平台统一 Key。

**原则**：登录不是装饰。只有当项目持久化升级到数据库后，登录才有意义。

### 2. 数据库持久化

**目标**：保存项目、章节、角色、场景和生成记录。

**建议数据表**：

- `users`
- `projects`
- `novel_chapters`
- `script_versions`
- `model_configs`
- `generation_runs`
- `validation_errors`

**原则**：先保存完整 YAML 版本，再考虑把每个 scene 拆表。不要一上来过度建模。

### 3. 管理端

**目标**：管理模型、用户、项目、调用成本和失败日志。

**建议模块**：

- 用户列表。
- 项目列表。
- 模型供应商配置。
- Prompt 模板管理。
- 调用日志。
- Token 消耗统计。
- Schema 校验失败统计。

### 4. 异步任务队列

**目标**：长小说转换不能阻塞 HTTP 请求。

**建议能力**：

- 提交转换任务。
- 后台分章处理。
- 页面轮询或 SSE 展示进度。
- 失败可重试。

## P3：真正拉开差距的 AI 能力

### 1. 角色一致性追踪

跨章节追踪角色别名、关系、性格变化，避免前后不一致。

输出：

- 角色小传。
- 角色关系图。
- 角色目标 / 阻碍 / 转变。

### 2. 故事结构分析

不仅转场景，还分析三幕式、起承转合、冲突强度。

输出：

- 故事节拍表。
- 关键冲突点。
- 高潮位置。
- 节奏问题提示。

### 3. 多版本生成对比

同一章节生成多个改编版本，用户选择更好的。

对比维度：

- 台词更自然。
- 冲突更强。
- 节奏更快。
- 更贴近原文。

### 4. 人机协作改编流程

让作者每一步都能控制，而不是 AI 一次性黑盒输出。

```text
小说输入
↓
章节摘要确认
↓
角色表确认
↓
场景拆分确认
↓
剧本生成
↓
局部修订
↓
导出
```

这比“一键生成”更像真正的创作工具。

## 不应该优先做的东西

- 多人实时协作：没有数据库项目系统前别碰。
- 复杂权限系统：没有管理端数据前别碰。
- 支付系统：比赛阶段没有意义。
- 模板市场：现在连模板质量都没验证。
- 过度数据库拆表：先保存 YAML 版本，别把结构搞死。
- 花哨动画：界面够清晰就行，不要掩盖核心能力不足。
