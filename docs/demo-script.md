# 现场演示口播稿

这份稿子可以直接照读。建议语速正常，控制在 5 到 7 分钟。

## 0. 开场

大家好，我演示的是题目三：AI 小说转剧本工具。

这个题目的核心不是让 AI 随便写一段剧情，而是帮助小说作者把 3 个章节以上的小说文本，转换成结构化的 YAML 剧本初稿。作者后续可以继续编辑、校验、保存和导出。

所以我这个项目围绕一条主线做：导入小说，识别章节，调用 OpenAI-compatible 模型生成剧本，再用 YAML Schema 校验，最后保存和导出。

## 1. 导入小说

现在我进入工作台。

这里可以导入 `.txt` 或 `.md` 小说文件。我选择样例小说 `samples/novel-3chapters.txt`。这个样例不是一两句话，而是 5 个章节的小说文本，满足题目要求的 3 章以上。

导入后，系统会在浏览器本地读取文件，并生成作品标题。这里不需要先上传到服务器，作者可以先在本地确认文本内容。

## 2. 章节大纲

下面是章节大纲预览。

可以看到系统识别出了每个章节的标题、字数和正文预览。这个步骤很重要，因为它让生成过程不再是黑盒。作者在转换之前就能确认：系统确实读到了 3 章以上内容，而且章节切分是正确的。

如果输入少于 3 章，系统会直接拒绝转换，不会生成假结果。

## 3. 模型配置

这里是模型配置。

项目只维护一套 OpenAI-compatible 配置，不做复杂的多供应商后台。原因是这个题目的核心是小说转剧本，不是 AI 平台管理系统。

在本地开发时，可以填写 base URL 和 API Key，然后点击获取模型，临时拉取可用模型列表，选择一个更便宜或更合适的 model。模型列表只是临时获取，不入库；真正保存的是当前选择的运行配置。生产环境则应该由服务端持有 API Key，不在浏览器暴露真实密钥。

## 4. 生成 YAML 剧本

现在点击“转换为 YAML 剧本”。

生成完成后，可以看到输出不是普通文本，而是 YAML。顶层结构包括 `metadata`、`characters`、`scenes` 和 `summary`。

`metadata` 记录作品标题、来源章节数、语言和 Schema 版本；`characters` 是角色表；`scenes` 是按场景组织的剧本正文；`summary` 是一句话梗概、主题和改编说明。

这就是题目要求的结构化剧本初稿。作者可以直接修改台词、动作、场景和镜头提示。

## 5. Schema 校验和质量清单

右侧这里是 Schema 校验和剧本质量清单。

Schema 校验通过，说明 YAML 不只是格式正确，而且字段结构符合我们定义的剧本 Schema。

质量清单会继续检查元信息、角色、场景、台词、角色引用和总结是否完整。这里的质量清单不是主观评价剧情好不好，而是判断这份剧本初稿是否具备可交付结构。

项目没有用空字符串去兜底缺失字段。如果 AI 少返回了 `metadata.title` 或者某个场景没有台词，系统会直接报错。这样可以避免作者拿到一份看起来完整、实际缺字段的坏剧本。

## 6. 入库和继续打磨

接下来我展示保存能力。

我点击“保存到项目库”。保存完成后进入项目库，可以看到项目卡片，里面展示了项目标题、最近生成状态和最近使用的模型。

这说明入库能力不是藏在后台的，它在产品里是可见的。作者可以从项目库找回之前的小说和 YAML 版本，继续打磨，而不是每次都重新导入和重新生成。

另外，工作台也支持浏览器本地草稿。我保存一个本地草稿，刷新页面后再加载，可以看到标题、小说正文、YAML 和转换报告都恢复了。删除草稿也不会清空当前编辑区，避免误操作。

## 7. 失败路径

我现在演示一个失败路径。

我手动删除 YAML 里的 `metadata.title`。可以看到 Schema 校验立刻失败，错误路径定位到 `metadata.title`，质量清单也出现失败项，同时导出按钮不可用。

这说明系统不会把坏 YAML 当成正常结果交给作者。恢复字段之后，校验重新通过，导出能力恢复。

## 8. 导出和 Schema 文档

现在点击导出 YAML。作者就能得到一份 `.yaml` 剧本文件，后续可以继续编辑、存档或交给其他工具处理。

最后我打开 `docs/script-yaml-schema.md`。这份文档定义了 YAML Schema，包括 `metadata`、`characters`、`scenes`、`dialogue`、`summary` 的字段规则，并说明了每个结构的设计原因。

比如为什么用 `scenes` 而不是直接沿用小说章节：因为剧本的最小工作单元是场景；为什么角色使用 `id`：因为场景和台词需要稳定引用，避免同名和改名造成混乱。

## 9. 结尾

总结一下，这个项目已经完成题目三的闭环：它能导入 3 个章节以上的小说文本，调用 AI 转换成结构化 YAML 剧本，提供 Schema 文档、实时校验、质量清单、项目库保存、本地草稿和 YAML 导出。

作者拿到的不是一段不可控的 AI 文本，而是一份可以继续编辑、校验和打磨的剧本初稿。

## 现场异常备用说法

如果真实模型接口现场不稳定，可以这样说：

现场 AI 接口返回失败时，系统不会伪造成功结果，也不会用默认字段兜底。它会把错误暴露出来，避免作者导出坏 YAML。为了继续演示结构闭环，我会使用样例输出展示通过 Schema 的剧本结构、编辑校验、保存和导出能力。
